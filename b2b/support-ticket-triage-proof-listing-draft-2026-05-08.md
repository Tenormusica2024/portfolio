## 目的

B2Bポートフォリオへ掲載するか判断するための文案ドラフトです。
このファイルは掲載判断用であり、B2B本番UIにはまだ反映していません。

## 掲載タイトル候補

問い合わせ・保守依頼の一次整理 AI秘書Proof

## 1行説明

問い合わせや保守依頼をサンプルデータで分類し、緊急対応・返信下書き・後回し・ブロック中に分け、外部更新は確認キューへ回すAI秘書Proofです。

## 掲載カード本文案

問い合わせや保守依頼をAIが直接更新するのではなく、まず「緊急対応」「返信が必要」「後回し」「ブロック中」「対応不要」に整理する公開Proofです。サンプルチケットをもとに、緊急度・影響度・返信要否を判定し、必要な返信はdraft-onlyで作成します。コメント投稿、ラベル変更、チケット更新、通知送信など外部に影響する操作は自動実行せず、確認キューに入れて人が承認する前提にしています。

## B2B向け訴求ポイント

- 問い合わせ・保守依頼・作業依頼の一次整理を説明しやすい
- AIが勝手に返信・更新するのではなく、人間確認前提の導線を示せる
- 緊急度と影響度を分けて、対応優先度を説明できる
- draft-only reply と confirmation queue の安全境界が分かりやすい
- 公開repoはサンプルデータのみで、実顧客情報・実チケット・内部ログを含まない

## 成果指標候補

- 問い合わせを緊急 / 返信必要 / 後回し / ブロック中 / 対応不要へ分けられるか
- 返信が必要なものだけdraft-onlyで作れるか
- 外部更新が勝手に実行されず、確認キューに入るか
- 判断理由がスコア・カテゴリ・次アクションとして説明できるか
- サンプルデータだけで安全にデモできるか

## 担当範囲として見せる内容

- support intake / inquiry intake の安全境界設計
- サンプルチケット設計
- 優先度・影響度スコアリング設計
- draft-only reply 設計
- confirmation queue 設計
- no-comment / no-label-change / no-ticket-update の運用ガード
- public proof化に向けたREADME、CI、release整備

## 掲載時の注意

- 「自動で返信する」「自動でチケット更新する」とは書かない
- 「一次整理」「下書き」「確認キュー」「人間承認前提」の表現に寄せる
- 実運用データではなく、公開用サンプルデータであることを明記する
- B2B本番UIへ載せる場合は、desktop / mobile / 実URLで目視確認してから反映する

## 短縮版コピー

問い合わせや保守依頼をAIが直接更新するのではなく、まず緊急対応・返信必要・後回し・ブロック中に分類する公開Proofです。返信案はdraft-onlyで作成し、コメント投稿やチケット更新など外部に影響する操作は確認キューへ回すことで、安全に使えるサポート業務導入パターンを示しています。

## CTA近くに置く補足文案

AI導入で重要なのは、問い合わせ対応をいきなり完全自動化することではなく、AIが整理する部分と人が承認すべき操作を分けることです。このProofでは、外部更新を確認キューへ分離し、サポート業務に入れやすい安全境界を示しています。

## 次にUIへ載せる場合の最小手順

1. この文案から掲載本文を3〜5文に圧縮する。
2. daily / Gmail / support ticket の3 proofを同じセクションに並べるか判断する。
3. UI変更前に desktop / mobile の見え方を想定して短いセクション案を作る。
4. 実装後、local と production URL の両方を目視確認する。
5. 問い合わせCTAへの導線が過剰にならないよう、リンクは1箇所に絞る。
